# Data Model & State Machines

**Feature**: Autonomous Specification Execution System
**Date**: 2025-10-27
**Phase**: 1 - Design

## Primary Entities

### 1. Specification

Represents a complete feature with tasks, acceptance criteria, and metadata.

**Attributes**:
```typescript
interface Specification {
  id: string;                    // Unique identifier (e.g., "003-orchestrator-agents")
  title: string;                 // Human-readable name
  status: SpecStatus;            // Current state: draft | in_progress | completed | failed
  created: string;               // ISO-8601 timestamp
  updated: string;               // ISO-8601 timestamp
  priority: 'critical' | 'high' | 'medium' | 'low';
  assignee?: string;             // Optional: which agent/human owns this
  tasks: Task[];                 // Ordered list of work items
  acceptanceCriteria: AcceptanceCriterion[];
  qaRules?: QARule[];           // Optional: Q&A patterns for this spec
}

type SpecStatus = 'draft' | 'in_progress' | 'completed' | 'failed';
```

**Relationships**:
- Contains many **Tasks** (1:N)
- Has many **AcceptanceCriteria** (1:N)
- Has many optional **QARule**s (1:N)

**Validation Rules**:
- `id` must be unique across repository
- `id` must match pattern: `\d{3,}-[\w-]+`
- `tasks` must have at least one entry for active specs
- `created` must be ≤ `updated`
- `status` transitions must follow state machine (see below)

**Source**: `.specify/specs/{id}/spec.md` (GitHub Spec Kit format)

### 2. Task

Individual work item within a specification, with dependencies and status tracking.

**Attributes**:
```typescript
interface Task {
  id: string;                    // Unique within spec (e.g., "T001")
  specId: string;                // Parent specification ID
  description: string;           // What needs to be done
  status: TaskStatus;            // Current state
  deliveryPrompt?: string;       // Instructions for AI assistant
  dependencies: string[];        // Task IDs that must complete first
  attemptCount: number;          // Retry tracking (0-3)
  lastError?: string;            // Most recent failure reason
  startedAt?: string;            // ISO-8601 timestamp
  completedAt?: string;          // ISO-8601 timestamp
}

type TaskStatus = 'pending' | 'in_progress' | 'completed' | 'failed';
```

**Relationships**:
- Belongs to one **Specification** (N:1)
- Depends on zero or more **Task**s (N:N self-referential)
- Generates one or more **ValidationResult**s during execution (1:N)
- Generates one or more **TestResult**s during execution (1:N)

**Validation Rules**:
- `id` unique within `specId`
- `dependencies` must reference valid task IDs in same spec
- Circular dependencies not allowed (detected via topological sort)
- `attemptCount` must be 0-3 (escalate to human at 3)
- `status` transitions must follow state machine (see below)

**Dependency Resolution**:
- Task cannot start unless all `dependencies` have `status === 'completed'`
- Orchestrator builds topological sort of task graph at startup
- Tasks with no dependencies start immediately

**Source**: `.specify/specs/{specId}/tasks.md` (checkbox lists with `deps:` annotations)

### 3. ValidationResult

Assessment of code quality from Engineer Agent after implementation.

**Attributes**:
```typescript
interface ValidationResult {
  id: string;                    // Unique identifier
  taskId: string;                // Which task this validates
  timestamp: string;             // ISO-8601 timestamp
  isValid: boolean;              // Overall pass/fail
  issues: ValidationIssue[];     // List of problems found
  suggestions: string[];         // Actionable recommendations
  constitutionChecks: Record<string, boolean>; // Per-principle compliance
}

interface ValidationIssue {
  category: 'functional' | 'security' | 'performance' | 'quality' | 'constitution';
  severity: 'blocker' | 'critical' | 'major' | 'minor';
  description: string;
  location?: string;             // File:line reference
}
```

**Relationships**:
- Belongs to one **Task** (N:1)
- References **Constitution** principles (implicitly)

**Validation Rules**:
- `isValid` is false if any `issue.severity === 'blocker' | 'critical'`
- `constitutionChecks` must include all 7 principles (I-VII)
- `suggestions` should map 1:1 with `issues` when possible

**Source**: Generated by EngineerAgent.validate() via Claude API

### 4. TestResult

Output from running automated tests (unit, integration, E2E).

**Attributes**:
```typescript
interface TestResult {
  id: string;                    // Unique identifier
  taskId: string;                // Which task this tests
  timestamp: string;             // ISO-8601 timestamp
  testType: 'unit' | 'integration' | 'e2e';
  passed: boolean;               // Overall pass/fail
  totalTests: number;
  passedTests: number;
  failedTests: number;
  skippedTests: number;
  duration: number;              // Milliseconds
  failures: TestFailure[];       // Details of failed tests
  coverage?: CoverageReport;     // Optional: coverage metrics
}

interface TestFailure {
  testName: string;
  error: string;
  stackTrace?: string;
  screenshot?: string;           // For E2E tests (Playwright)
}

interface CoverageReport {
  line: number;                  // Percentage (0-100)
  branch: number;
  function: number;
  statement: number;
}
```

**Relationships**:
- Belongs to one **Task** (N:1)

**Validation Rules**:
- `passed` is false if `failedTests > 0`
- `passedTests + failedTests + skippedTests === totalTests`
- `coverage` (if present) must have all metrics ≥80% for constitutional compliance
- `duration` must be reasonable per test type (unit: <2min, integration: <3min, e2e: <5min)

**Source**: Generated by TestAgent.runTests() via Playwright/Vitest execution

### 5. Question

Inquiry from AI assistant about requirements, needing clarification.

**Attributes**:
```typescript
interface Question {
  id: string;                    // Unique identifier
  taskId?: string;               // Optional: which task prompted this
  specId: string;                // Which spec this relates to
  timestamp: string;             // ISO-8601 timestamp
  question: string;              // The actual question text
  context: string;               // What the agent was working on
  answeredBy?: 'qa-engine' | 'human';
  answerId?: string;             // Reference to Answer entity
}
```

**Relationships**:
- May belong to one **Task** (N:1 optional)
- Belongs to one **Specification** (N:1)
- Has zero or one **Answer** (1:1)

**Validation Rules**:
- `question` must not be empty
- If `answeredBy` is set, `answerId` must also be set
- If escalated to human, create **Notification**

**Source**: Detected by ClaudeCodeInterceptor monitoring `.claude-question.txt`

### 6. Answer

Response to a question, with confidence scoring.

**Attributes**:
```typescript
interface Answer {
  id: string;                    // Unique identifier
  questionId: string;            // Which question this answers
  timestamp: string;             // ISO-8601 timestamp
  answer: string;                // The actual answer text
  confidence: number;            // 0-100 percentage
  sources: string[];             // Which specs were used to derive answer
  method: 'qa-engine' | 'human'; // How answer was obtained
}
```

**Relationships**:
- Belongs to one **Question** (1:1)
- References zero or more **Specification**s as sources (N:N)

**Validation Rules**:
- `confidence` must be 0-100
- If `confidence < 80`, must escalate to human (per FR-012)
- `sources` should list spec IDs that contributed to answer
- `method === 'human'` implies `confidence === 100`

**Source**: Generated by QAEngine.answerQuestion() via Claude API or human input

### 7. Notification

Alert sent to humans when orchestrator needs intervention.

**Attributes**:
```typescript
interface Notification {
  id: string;                    // Unique identifier
  timestamp: string;             // ISO-8601 timestamp
  severity: 'info' | 'warning' | 'critical';
  message: string;               // Human-readable summary
  context: NotificationContext;  // Structured details
  deliveryStatus: 'queued' | 'sending' | 'delivered' | 'failed' | 'persisted';
  deliveryMethod: 'whatsapp' | 'file-fallback';
  deliveryError?: string;        // If failed, why?
  respondedAt?: string;          // If human responded, when?
  response?: string;             // If human responded, what?
}

interface NotificationContext {
  taskId?: string;
  specId: string;
  reason: 'task-failed' | 'question-escalated' | 'critical-error';
  details: Record<string, any>;  // Flexible context (e.g., error stack, question text)
}
```

**Relationships**:
- May relate to one **Task** (N:1 optional)
- Relates to one **Specification** (N:1)
- May relate to one **Question** (N:1 optional)

**Validation Rules**:
- `severity === 'critical'` requires immediate delivery attempt
- If `deliveryStatus === 'failed'`, must write to `.specify/.notifications.log` (FR-019)
- `context.reason` must match one of predefined escalation scenarios

**Source**: Generated by NotificationService.sendSMS() or Orchestrator on escalation

### 8. LogEntry (NEW)

Structured operational log entry for monitoring and debugging.

**Attributes**:
```typescript
interface LogEntry {
  timestamp: string;             // ISO-8601 timestamp
  level: 'INFO' | 'WARN' | 'ERROR';
  event: string;                 // Event type (see R1 in research.md)
  taskId?: string;               // Optional: related task
  specId?: string;               // Optional: related spec

}
```

**Relationships**:
- May relate to **Task** (implicit, via `taskId`)
- May relate to **Specification** (implicit, via `specId`)

**Validation Rules**:
- `timestamp` must be ISO-8601 format
- `event` must be from predefined event taxonomy (see R1)
- `level` determines routing (ERROR → `.specify/.orchestrator.log` file)

**Output**: stdout (JSON lines) and `.specify/.orchestrator.log` (WARN/ERROR only)

## State Machines

### Task Status State Machine

```
┌─────────┐
│ pending │ Initial state (not started)
└────┬────┘
     │ Orchestrator.executeTask()
     ▼
┌──────────────┐
│ in_progress  │ Task executing (prompt sent to Claude Code)
└──┬────────┬──┘
   │        │
   │        │ Tests pass + validation passes
   │        ▼
   │   ┌───────────┐
   │   │ completed │ Terminal state (success)
   │   └───────────┘
   │
   │ Tests fail OR validation fails
   ▼
┌────────┐
│ failed │ Terminal state (after 3 attempts → human escalation)
└────────┘
```

**Transition Rules**:
- `pending → in_progress`: When dependencies satisfied and orchestrator selects task
- `in_progress → completed`: When tests pass AND validation passes
- `in_progress → failed`: When attempt count reaches 3 OR critical error occurs
- `in_progress → in_progress` (self-loop): Retry with feedback (attempts 1, 2)
- No transitions FROM `completed` or `failed` (terminal states)

**Triggers**:
- `pending → in_progress`: Orchestrator.executeTask()
- `in_progress → completed`: TestAgent.runTests() success + EngineerAgent.validate() success
- `in_progress → failed`: attemptCount >= 3 OR critical error
- Retry: Orchestrator.retryTaskExecution() after exponential backoff

---

### Specification Status State Machine

```
┌───────┐
│ draft │ Initial state (spec created, not started)
└───┬───┘
    │ First task starts
    ▼
┌─────────────┐
│ in_progress │ At least one task executing or completed
└──┬────────┬─┘
   │        │
   │        │ All tasks completed
   │        ▼
   │   ┌───────────┐
   │   │ completed │ Terminal state (all tasks done)
   │   └───────────┘
   │
   │ Any task failed AND no pending tasks
   ▼
┌────────┐
│ failed │ Terminal state (requires human intervention)
└────────┘
```

**Transition Rules**:
- `draft → in_progress`: When first task moves to `in_progress`
- `in_progress → completed`: When all tasks have `status === 'completed'`
- `in_progress → failed`: When any task has `status === 'failed'` AND no `pending` tasks remain
- No transitions FROM `completed` or `failed` (terminal states)

---

### Notification Delivery State Machine

```
┌────────┐
│ queued │ Initial state (notification created)
└────┬───┘
     │ NotificationService.send()
     ▼
┌─────────┐
│ sending │ WhatsApp API call in progress
└──┬────┬─┘
   │    │
   │    │ WhatsApp API success
   │    ▼
   │ ┌───────────┐
   │ │ delivered │ Terminal state (success)
   │ └───────────┘
   │
   │ WhatsApp API failure
   ▼
┌────────┐
│ failed │ Intermediate state (attempt fallback)
└───┬────┘
    │ Write to .notifications.log
    ▼
┌───────────┐
│ persisted │ Terminal state (fallback successful)
└───────────┘
```

**Transition Rules**:
- `queued → sending`: When NotificationService.send() called
- `sending → delivered`: When WhatsApp API returns success
- `sending → failed`: When WhatsApp API throws error OR timeout
- `failed → persisted`: When `.specify/.notifications.log` write succeeds
- If `.notifications.log` write fails, log ERROR and remain in `failed` (catastrophic)

---

### Orchestrator State Machine

```
┌──────────┐
│ starting │ Initial state (loading specs, building queue)
└────┬─────┘
     │ Specs loaded + queue built
     ▼
┌─────────┐
│ running │ Main loop (processing tasks)
└──┬────┬─┘
   │    │
   │    │ User SIGINT/SIGTERM
   │    ▼
   │ ┌─────────┐
   │ │ stopped │ Terminal state (graceful shutdown)
   │ └─────────┘
   │
   │ Critical error (e.g., file system failure)
   ▼
┌────────┐
│ paused │ Intermediate state (waiting for recovery or human)
└────┬───┘
     │ Manual restart OR error resolved
     ▼
   (back to starting)
```

**Transition Rules**:
- `starting → running`: When all specs loaded and task queue built successfully
- `running → stopped`: When SIGINT/SIGTERM received OR all specs completed
- `running → paused`: When critical error occurs (e.g., `.specify/` directory unavailable)
- `paused → starting`: When orchestrator restarted manually (no automatic recovery in MVP)

## Entity Persistence

| Entity | Storage Location | Format | Mutability |
|--------|------------------|--------|------------|
| **Specification** | `.specify/specs/{id}/spec.md` | GitHub Spec Kit (YAML + Markdown) | Updated by orchestrator (status, tasks) |
| **Task** | `.specify/specs/{id}/tasks.md` | Checkbox lists with `deps:` | Updated by orchestrator (status checkboxes) |
| **ValidationResult** | In-memory only | N/A | Ephemeral (used for retry logic) |
| **TestResult** | In-memory only + logs | JSON in stdout logs | Ephemeral (test reports in CI) |
| **Question** | `.claude-question.txt` (input) | Plain text | Read-once, cleared after processing |
| **Answer** | `.claude-output.txt` (response) | Plain text | Read-once, cleared after processing |
| **Notification** | `.specify/.notifications.log` | JSON lines (fallback) | Append-only |
| **LogEntry** | stdout + `.specify/.orchestrator.log` | JSON lines | Append-only |

**Write Strategy**:
- Spec/Task updates use last-write-wins (per R2)
- Notification log is append-only (never truncated)
- Communication files (`.claude-*.txt`) are write-once, clear-on-read

**Concurrency**:
- Single-process orchestrator (no multi-process concerns)
- File watchers handle external edits (Chokidar)
- Spec file modifications detected via mtime comparison

## Data Model Summary

**8 Core Entities**:
1. Specification (persistent)
2. Task (persistent)
3. ValidationResult (ephemeral)
4. TestResult (ephemeral)
5. Question (transient, file-based)
6. Answer (transient, file-based)
7. Notification (semi-persistent, log-based)
8. LogEntry (persistent, log-based)

**4 State Machines**:
1. Task Status (pending → in_progress → completed/failed)
2. Specification Status (draft → in_progress → completed/failed)
3. Notification Delivery (queued → sending → delivered/failed/persisted)
4. Orchestrator State (starting → running → stopped/paused)

**Key Relationships**:
- Specification 1:N Task
- Task 1:N ValidationResult
- Task 1:N TestResult
- Specification/Task 1:N Question
- Question 1:1 Answer
- Task/Spec 1:N Notification
