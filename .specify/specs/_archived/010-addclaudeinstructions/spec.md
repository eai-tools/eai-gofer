# Default AI Instruction Files for New Repositories

## Overview

When Gofer initializes a new repository, it currently creates
`.claude/commands/`, `.claude/agents/`, `.github/prompts/`,
`.github/instructions/`, and `.specify/` directory structures but does NOT
generate AGENTS.md, CLAUDE.md, or `.github/copilot-instructions.md`. This means
AI coding assistants have no project-specific context when working in the
repository.

This feature adds smart, template-based generation of three instruction files
during Gofer initialization, following the 2026 industry-standard three-tier
content architecture:

1. **AGENTS.md** - Cross-tool universal base (works with Claude, Copilot,
   Cursor, Windsurf, Aider, and 20+ tools)
2. **CLAUDE.md** - Claude Code-specific extensions with `@AGENTS.md` import
3. **`.github/copilot-instructions.md`** - GitHub Copilot-specific extensions

The files are generated by detecting project characteristics (language,
framework, test runner, build system) and assembling appropriate template
sections. Files are never overwritten if they already exist.

**Research Reference**: See `research.md` for market analysis, codebase
patterns, and technology decisions.

## User Stories

### US1: Initialize AI Instructions in New Repository (P1)

**As a** developer setting up Gofer in a new repository **I want** AGENTS.md,
CLAUDE.md, and copilot-instructions.md to be auto-generated **So that** AI
coding assistants immediately have project context regardless of which tool I
use

**Acceptance Criteria**:

- [ ] Running "Gofer: Initialize Repository" creates all three instruction files
      at the correct locations (AGENTS.md and CLAUDE.md at workspace root,
      `.github/copilot-instructions.md` in `.github/` directory)
- [ ] Generated AGENTS.md contains detected project commands, structure, code
      style, testing instructions, git workflow, and boundaries
- [ ] Generated CLAUDE.md is under 60 lines and uses `@AGENTS.md` import for
      shared content
- [ ] Generated copilot-instructions.md includes project overview and available
      Gofer commands
- [ ] No file is created if it already exists in the workspace
- [ ] The generation works without an AI API key (purely deterministic template
      assembly)

### US2: Project-Aware Template Selection (P1)

**As a** developer working in a TypeScript, Python, Go, or other language
project **I want** the generated instruction files to match my project's
technology stack **So that** the AI assistant receives relevant conventions, not
generic ones

**Acceptance Criteria**:

- [ ] Project detection identifies language from manifest files (package.json,
      tsconfig.json, pyproject.toml, go.mod, Cargo.toml, pom.xml, build.gradle)
- [ ] Detected test runner is included in AGENTS.md commands section (e.g.,
      `npm test`, `pytest`, `go test`)
- [ ] Detected build commands are included in AGENTS.md commands section
- [ ] Detected linting/formatting tools are referenced in code style section
- [ ] If no project files are detected, a minimal generic template is generated

### US3: Workflow Orchestration Principles in Instructions (P1)

**As a** developer using Gofer's pipeline **I want** my workflow principles
embedded in the generated instruction files **So that** AI assistants follow my
preferred working style (plan-first, verification, elegance, autonomous bug
fixing)

**Acceptance Criteria**:

- [ ] CLAUDE.md includes brief workflow guidance: plan mode for non-trivial
      tasks, re-plan on failure, update lessons.md after corrections, demand
      elegance, autonomous bug fixing
- [ ] AGENTS.md includes core principles: simplicity first, find root causes,
      minimal impact, verify before marking complete
- [ ] Gofer pipeline commands are referenced in CLAUDE.md (available commands
      section)
- [ ] Copilot-instructions.md references available Gofer prompts

### US4: Regenerate Instructions After Changes (P2)

**As a** developer whose project has evolved (new framework, changed test
runner) **I want** to regenerate instruction files with updated project
detection **So that** the AI instructions stay current with my project

**Acceptance Criteria**:

- [ ] A "Gofer: Regenerate Instructions" VS Code command is available
- [ ] If files already exist, user is prompted with options: overwrite, skip, or
      backup & replace
- [ ] Regeneration re-detects project characteristics from current state
- [ ] The command can be run from the VS Code Command Palette

### US5: Existing Installation Sync (P2)

**As a** developer with an existing Gofer installation that predates this
feature **I want** to be informed about missing instruction files during upgrade
**So that** I can opt in to generating them without disruption

**Acceptance Criteria**:

- [ ] During `syncMissingResources()`, missing instruction files are detected
- [ ] User is prompted: "Missing AI instruction files (AGENTS.md, CLAUDE.md).
      Generate them?"
- [ ] If user declines, no files are created and no further prompts occur for
      that session
- [ ] Existing files are never modified or overwritten during sync

## Functional Requirements

### FR1: Three-Tier Content Architecture

The system MUST generate three instruction files following the industry-standard
content hierarchy:

- **AGENTS.md** (workspace root): Cross-tool universal base containing commands,
  project structure, code style, testing, git workflow, and boundaries. Target
  size: 100-150 lines.
- **CLAUDE.md** (workspace root): Claude Code-specific file using `@AGENTS.md`
  import for shared content, plus Gofer pipeline commands, workflow principles,
  and context management hints. Target size: 40-60 lines.
- **`.github/copilot-instructions.md`**: Copilot-specific file with project
  overview, available Gofer prompts, and condensed code quality rules. Target
  size: 80-100 lines.

**Validation**: Measure generated file line counts; assert within target ranges.
**Integration**: Uses ResourceSyncer.copyBundledResources() pattern from
`ResourceSyncer.ts:61`.

### FR2: Project Detection Engine

The system MUST detect project characteristics by scanning for manifest files:

| Manifest File                                      | Detects                                               |
| -------------------------------------------------- | ----------------------------------------------------- |
| `package.json`                                     | Node.js/JavaScript project, npm scripts, dependencies |
| `tsconfig.json`                                    | TypeScript project                                    |
| `pyproject.toml` / `setup.py` / `requirements.txt` | Python project                                        |
| `go.mod`                                           | Go project                                            |
| `Cargo.toml`                                       | Rust project                                          |
| `pom.xml` / `build.gradle`                         | Java project                                          |
| `.eslintrc*` / `eslint.config.*`                   | ESLint configuration                                  |
| `.prettierrc*`                                     | Prettier configuration                                |
| `vitest.config.*` / `jest.config.*`                | Test runner                                           |
| `Makefile`                                         | Make build system                                     |

Detection MUST work without network access or API keys.

**Validation**: Unit test with mock filesystem containing various project types.
**Integration**: New ProjectDetector class, used by ResourceSyncer.

### FR3: Template Assembly System

The system MUST assemble instruction files from composable template fragments:

- Base template for each file (AGENTS.md, CLAUDE.md, copilot-instructions.md)
- Language-specific fragments (TypeScript, Python, Go, etc.)
- Framework-specific fragments (React, Express, Django, etc.)
- Tool-specific fragments (ESLint, Prettier, Vitest, etc.)
- Gofer-specific fragment (pipeline commands, workflow principles)

Template fragments are stored in `extension/resources/instruction-templates/`
and bundled with the extension.

**Validation**: Generate templates for TypeScript, Python, and Go projects;
verify correct fragment assembly. **Integration**: New
InstructionTemplateAssembler class.

### FR4: Safe File Creation

The system MUST check for existing files before creating instruction files.

- If file exists: skip creation, log that file was preserved
- If file does not exist: create from template assembly
- `.github/` directory MUST be created if it doesn't exist (for
  copilot-instructions.md)
- Use FileUtils.exists() and FileUtils.writeTextFile() from existing utilities

**Validation**: Test with pre-existing files; verify no overwrite.
**Integration**: Uses FileUtils from `extension/src/utils/fileUtils.ts`.

### FR5: Upgrade Service Integration

The system MUST integrate into the existing UpgradeService upgrade sequence:

- New step added after `setupCopilotInstructions()` (line 209 of
  UpgradeService.ts)
- Method signature: `setupDefaultInstructions(): Promise<void>` added to
  IResourceOperations interface
- Implementation in ResourceSyncer
- Also called from `syncMissingResources()` for existing installations

**Validation**: Integration test verifying step runs during upgrade flow.
**Integration**: UpgradeService.ts:209, IResourceOperations interface,
ResourceSyncer, GoferMigrator.syncMissingResources().

### FR6: Regenerate Instructions Command

The system MUST provide a VS Code command "Gofer: Regenerate Instructions"
(`gofer.regenerateInstructions`) that:

- Re-runs project detection on current workspace
- Prompts user for each existing file: overwrite, skip, or backup-and-replace
- Generates new files based on current project state
- Reports results to user via VS Code notification

**Validation**: Manual test via Command Palette. **Integration**:
CommandRegistry, registered in registerGlobalCommands().

### FR7: Content Partitioning

The system MUST correctly partition content between instruction files and Gofer
pipeline commands:

**In instruction files (persistent, loaded every conversation):**

- Code style rules and conventions
- Build, test, lint, format commands
- Project structure description
- Git workflow conventions
- Boundary rules (don't touch these files)
- Core principles (simplicity, root cause, minimal impact, verification)
- Brief workflow guidance (plan-first, elegance, autonomous bug fixing)

**In pipeline commands only (invoked on demand):**

- Detailed workflow orchestration (step-by-step procedures)
- Subagent coordination strategies
- Self-improvement loop mechanics (full procedure)
- Multi-step validation protocols
- Task management procedures (todo.md/lessons.md detailed workflow)

**Validation**: Review generated content; verify no procedural workflows in
instruction files.

## Non-Functional Requirements

### Performance

- Project detection MUST complete in under 2 seconds for a typical project
  (scanning manifest files only, not full directory tree)
- Template assembly MUST complete in under 500ms
- File creation MUST not block VS Code extension activation

### Compatibility

- Generated AGENTS.md MUST be valid for Claude Code, GitHub Copilot, Cursor,
  Windsurf, and Aider (no tool-specific syntax in AGENTS.md)
- Generated CLAUDE.md MUST use valid `@` import syntax
- Generated copilot-instructions.md MUST use valid Markdown without requiring
  YAML frontmatter (frontmatter is optional)
- Files MUST work on Windows, macOS, and Linux (no symlinks, LF line endings)

### Maintainability

- Template fragments MUST be separate files in `extension/resources/`, not
  hardcoded TypeScript strings
- Adding support for a new language MUST require only adding a new template
  fragment file, not modifying core logic
- Template fragments MUST be independently testable

## Assumptions

- Claude Code supports the `@AGENTS.md` import syntax for referencing other
  Markdown files from CLAUDE.md (confirmed by Anthropic docs and research)
- GitHub Copilot reads `.github/copilot-instructions.md` automatically when
  present (confirmed by GitHub docs)
- AGENTS.md is read by Claude Code, Copilot, Cursor, and other tools when
  present at workspace root (confirmed by Agentic AI Foundation)
- The ResourceSyncer.copyBundledResources() pattern can be adapted for
  template-based generation (not just static file copying)
- Windows users cannot rely on symlinks; each file must contain its own content
  or use tool-native import mechanisms
- The existing IResourceOperations interface can be extended with a new method
  without breaking backward compatibility (it's internal to the extension)

## Dependencies

- **ResourceSyncer** (`extension/src/services/migration/ResourceSyncer.ts`):
  Implements file creation following existing patterns
- **UpgradeService** (`extension/src/services/migration/UpgradeService.ts`):
  Orchestrates the new step in the upgrade sequence
- **IResourceOperations interface**: Must be extended with
  `setupDefaultInstructions()` method
- **FileUtils** (`extension/src/utils/fileUtils.ts`): Used for safe file
  operations (exists, ensureDirectory, writeTextFile)
- **GoferMigrator.syncMissingResources()**: Must include instruction file sync
  for existing installations
- **CommandRegistry**: Registers the new regenerate command

## Out of Scope

- AI-powered dynamic content generation (requires API key, unpredictable)
- Cursor `.cursorrules` or `.cursor/rules/*.mdc` generation (Cursor-specific
  format, limited adoption compared to AGENTS.md)
- Windsurf `.windsurfrules` generation (same rationale)
- Automatic updates to instruction files when project dependencies change
  (manual regenerate command is sufficient)
- Constitution integration (constitution.md is a separate Gofer concept;
  instruction files are for AI tool configuration)
- Language-specific nested AGENTS.md files in subdirectories (future
  enhancement, not MVP)
